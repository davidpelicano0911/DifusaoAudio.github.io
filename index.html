Contexto

A gestão eficiente de áudio é essencial em espaços como hotéis, centros comerciais, restaurantes e lojas onde diferentes zonas requerem anúncios personalizados ou uma simples música de ambiente. Este projeto pretende responder a estas necessidades através de um sistema wireless composto por um nó central de controlo e múltiplos recetores distribuidos. Para isso,temos como objetivo aumentar a flexibilidade na gestão de áudio, permitindo uma experiência adaptada a cada zona e garantindo qualidade, simplicidade e eficiência na operação.


Motivação

A gestão de múltiplas zonas com diferentes necessidades sonoras, quando feita através de sistemas tradicionais baseados em cabos, é frequentemente complexa e pouco flexível. O uso de cabos exige infraestrutura física extensa, instalação trabalhosa e custos elevados de manutenção. Este projeto propõe uma solução wireless eliminando a necessidade de cabeamento. Assim, é possível assegurar uma distribuição de áudio intuitiva e com maior facilidade para expansões ou reconfigurações.

Tecnologias Utilizadas



HTML, CSS e JavaScript utilizados na construção da interface web. Foram escolhidos por serem ferramentas padrão para interfaces web, executáveis em qualquer navegador. Esta abordagem permite ao utilizador controlar a transmissão de áudio ajustando a música para as zonas desejadas.

Python 3 e Flask utilizados para o desenvolvimento do servidor e lógica de controlo do sistema. Python 3 escolhido pela sua simplicidade e vasto ecossistema de bibliotecas, e Flask como framework web leve e eficiente para desenvolver rapidamente a interface de controlo e os serviços de backend do sistema.

Utilizamos o Yt-dlp para extrair o áudio de fontes online (como o YouTube), enquanto o Opus é um codec open source que escolhemos para ser responsável pela compressão eficiente do áudio antes da transmissão. No lado do recetor, o FFplay é usado para descodificar e reproduzir diretamente os fluxos RTP/Opus.

UDP Broadcast é utilizado inicialmente para anunciar a presença do servidor e permitir que os recetores (colunas) se conectem automaticamente. Após o emparelhamento, a transmissão de áudio é feita através de UDP Multicast, garantindo uma distribuição eficiente do fluxo de áudio apenas para os recetores conectados.

Distribuição de Áudio na Rede


O sistema envia áudio via Wi-Fi através de UDP Broadcast e Multicast. Os recetores são detetados automaticamente, recebem configurações como canal e volume, e são atualizados dinamicamente. A arquitetura garante comunicação em tempo real com base numa matriz de configuração personalizada para cada recetor.

Testes Realizados

Para validar o desempenho do sistema de difusão de áudio, foram realizados testes com diferentes valores de frame duration. O gráfico acima apresenta a percentagem de perda de pacotes (%) em função do número de canais utilizados (c), para cada configuração de frame duration.

A frame duration influencia indiretamente o tamanho dos pacotes ao definir o intervalo de tempo entre os envios. Frames mais curtos resultam em um maior número de pacotes por segundo, aumentando o tráfego na rede. Esse aumento de tráfego eleva significativamente a probabilidade de perda de pacotes à medida que o número de canais cresce. Por outro lado, frames mais longos (como 100 ms ou 120 ms) reduzem a frequência de envio, diminuindo a carga na rede. Isso resulta numa menor perda de pacotes mesmo com mais canais ativos. No entanto, observou-se que essas configurações ainda apresentaram perdas superiores à configuração com 80 ms. Isso ocorre devido ao limite de MTU (Unidade Máxima de Transmissão), que restringe o tamanho dos pacotes a 1500 bytes. Quando esse limite é ultrapassado, os pacotes são fragmentados, o que aumenta o tráfego e, consequentemente, a probabilidade de perda.

Esses resultados evidenciam um compromisso inerente entre latência e confiabilidade: frame durations curtas reduzem a latência do sistema, mas aumentam a vulnerabilidade a perdas em ambientes com muitos canais; já frames mais longos favorecem a estabilidade da transmissão, mas com maior risco de fragmentação e latência. Com base nessa análise, foi possível determinar configurações ideais de frame duration e número de canais, de forma a otimizar a qualidade do áudio transmitido e a escalabilidade do sistema de difusão.

Problemas Encontrados

Situação: Pacotes RTP chegavam com atraso aos recetores, comprometendo a sincronização.
Impacto: Alta latência, áudio com falhas. Transmissão limitada a 1 Mbps (2.4GHz) ou 6 Mbps (5GHz).
Solução: Configuração dos roteadores para usar apenas os modos 802.11n (2.4GHz) ou 802.11ac (5GHz), assegurando maior qualidade de transmissão.
